# 生成延迟优化方案(按实施阶段)

> 依据:docs/benchmark-3d.md 的实测 + 延迟分解(2026-07-09)。
> TRELLIS 单次生成分解:提交往返 0.5–1.6s / 排队 ~1.1s / 前处理 4–5s / 扩散采样 2–3s / 后处理(网格+烘焙+导出) 8–25s(波动大头) / GLB 下载 3–47s。
> 结论:排队不是瓶颈;采样已无肉;空间在前处理、后处理、下载三段。带贴图管线物理地板价 ≈12s。

## 阶段一:留在 fal(本周,demo 用这套)

按优先级:

1. **下载转存+压缩(收益最大,零风险)**
   fal 的 GLB 在美国 CDN,手机直连下载实测 3–47s 波动。改为:backend 生成完成后立即拉取 GLB(机房间快且稳),`gltf-transform` Draco 压缩(1.5MB→~0.4MB),从自己服务端下发。下载段 → 1–2s。
2. **预热(消灭冷启动 54–121s)**
   - backend 加 `GET /api/warmup?model=trellis`:朝 fal 发一张内置小图。
   - App 启动时立即调一次(评委打开 App→刷 feed→拍照,30s+ 足够焐热);
   - demo 开场前手动预热所有会用到的端点。
3. **上传前缩图**:客户端把相机原图缩到 1024px 再传(模型输入本来就这个量级),上传 2–5MB→~200KB。
4. **两段式出图(感知减半)**:拍照后同时发 Hunyuan 白模(~10s)和 TRELLIS(~20s),白模先给用户转着玩,贴图版到了无缝替换。
5. **并发双发对冲波动(可选,现场保险)**:同图同时发两个 TRELLIS 请求取先回者。共享卡波动 15–35s→大概率拿到 ~15s。成本翻倍(每次几美分),只在现场开。
6. **别轮询用推送**:fal 支持 webhook;backend 对 fal 用长连接/webhook,对 App 用 SSE/轮询间隔降到 0.5s。省平均 ~1s。

预期:端到端(拍照→可交互模型)从「15–35s + 下载 3–47s」→ **总计 ~15–25s,下载稳定 1–2s**。

## 阶段二:自部署 keep-warm(Plan B,彩排后决定)

- 触发条件:彩排时 fal 高峰波动不可接受(频繁 >30s)。
- 方案:Modal(serverless+keep_warm)或 RunPod(4090 ~$0.35–0.7/h)跑开源 TRELLIS,demo 时段独占。部署走 GitHub Actions,符合"本机只写代码"工作流。
- 额外优化:喂 backend 已抠好的白底图跳过管线抠图(-3s);砍掉预览视频渲染阶段(fal 日志里的 Rendering 段,纯浪费);贴图 512(烘焙减半)。
- 预期:**端到端 ~10–13s 且稳定**。成本:demo+彩排 5 小时 ≈$2–10。
- 注意:TRELLIS 环境难装(spconv/flash-attn),用社区现成 Docker 镜像。

## 评奖期(赛后几天~几周,评委零散体验)

- 用量零散 → 继续 fal 按次付费(余额不用不扣,充 $20–30 够几百次生成),**不要**为评奖期挂独占实例(纯闲置费)。
- 冷启动靠「打开 App 即预热」兜住;生成 UI 文案兜底:"正在唤醒 3D 引擎…"。

## 已否决的路

- 换更快的模型:Hunyuan turbo/mini-turbo 带贴图 22–26s 不比普通快(贴图是硬瓶颈);TRELLIS 降步 19–26s 无提升(采样只占 2–3s);SAM-3 42–45s 出局。**~20s(共享)/~12s(独占)就是带贴图的地板价。**
- 换平台减排队:排队实测仅 ~1.1s,无肉可省。
- 降低输入清晰度换速度:低清输入速度不变(但好消息:360p/7KB 的抖音级截帧质量几乎不掉,视频摘抄场景可放心用)。

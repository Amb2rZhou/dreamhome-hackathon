# 资产库优先:后端技术方案与 TODO(2026-07-15,v2 混合架构)

> 背景:团队决策从「Feed 流实时圈选生成」转向「资产库优先」——提前批量预制高质量 3D 资产入库,
> 保证 demo 可控;「刷一刷」圈选降级为素材筛选入口。本文是后端(周芷乐)的实施方案。
> 前置结论沿用 docs/benchmark-3d.md(TRELLIS 质量 6/6、热启动 ~20s)和 docs/latency-optimization.md。
> v2 变更:纯预计算 → **预计算 + 实时识别的混合架构**。
> 理由:纯预计算只对已入库视频生效,demo 若只能在手挑视频上演有"PPT 展示"嫌疑;
> 实时单帧检测(1–2s)技术上完全可行,补上它才是真实产品形态。
> v2.1 变更:「有没有人圈过」的判定改为**查溯源标注**(确定性,零误判);
> embedding 降级为建议性去重辅助,只提示、从不自动判定(长得像的资产太多,纯向量匹配容易错)。
> v2.2 变更:同款匹配的主依据改为**结构化标签**——3D 生成的同时由多模态识别产出标签,
> 后续圈选靠标签匹配命中即复用、不重新生成;**重复生成 = 标签不全的信号**,
> 合并资产并把标签取并集写全,匹配能力随使用自愈变准。

## 一、产品链路(后端视角)

新链路对后端提出四个能力:

1. **批量资产生产(离线)**:收集抖音家装视频 → 把视频里出现的所有家具变成 3D 资产(GLB)→ 按 空间/品类 入库。
2. **资产溯源**:每个资产记录它来自哪个视频、出现在哪些时间段。资产详情页可跳回原视频对应位置。
3. **暂停识别 + 高亮(在线)**:用户播放视频、在**任意一帧**暂停,画面里识别出的素材以光点闪烁;
   已入库的可点击进资产库,未入库的可圈选触发生成;也可「全选」把当前帧已入库素材一键加进"我的素材库"。
4. **"有没有人圈过"的判定分两层**:
   - **同视频内 = 查溯源标注**:每个资产都带 (video_id + 轨迹 + 时间段) 标注,暂停帧上某物体
     是否已入库,拿 (video_id, t, bbox) 去 track 索引查 IoU 重叠——确定性匹配,零误判。
   - **跨视频/新圈选 = 标签匹配**:资产在 3D 生成的同时产出结构化标签(品类/颜色/材质/风格/
     形态特征/尺寸档)。别人圈选同款时对抠图提取同一套标签,与库内匹配,命中(经用户确认)
     直接复用资产,**不重新生成 3D**。embedding 只作排序辅助,不做判定。
   - **自愈闭环**:同一物体被生成了两遍 = 上次匹配没召回 = 标签不全。处理:合并两个资产,
     标签取**并集**写全,track 全部指向保留的 asset——匹配能力越用越准。

暂停识别按视频状态分三层,这也是真实产品(Prime Video X-Ray、电商视频商品锚点)的标准做法:

| 场景 | 链路 | 延迟 |
|---|---|---|
| feed 内视频(平台已索引) | 查预计算时空索引 | 0(本地查表) |
| 未索引视频第一次在某帧暂停 | 实时单帧检测,结果**写回索引**(lazy indexing);此视频尚无标注 → 检测框全部按「没人圈过」显示,事实如此 | 1–2s(配扫描动画) |
| 圈选后库里无同款 | image-to-3D 生成 → 新资产入库 | ~20s(过程动画) |

## 二、核心设计 A:预计算时空索引(feed 内容,零延迟)

**难点**:画面会动,用户每次暂停的位置都不一样,怎么保证素材都被识别出来?

**答案:平台内容不做实时识别,做离线预计算。** 视频入库时跑一遍检测+追踪,把每个物体在
**每一帧**的位置(bbox/mask)算好存成索引;前端暂停在 t 秒,只是拿 t 去查表,毫秒级返回,零算力、零不确定性。
经济账也是这么算的:一个视频被暂停一万次,处理一次、查表一万次,而不是烧一万次 GPU。

离线 pipeline(每个视频跑一次):

```
视频文件
 ├─ 1. 抽帧 + 场景切分(镜头切换点,PySceneDetect 或帧差)
 ├─ 2. 开集检测:在关键帧上找家具(Grounding DINO / YOLO-World,
 │     词表:沙发/单椅/床/柜/桌/灯/地毯/绿植/窗帘/装饰画…)
 ├─ 3. SAM2 视频追踪:每个检测框作为 prompt,SAM2 跨帧追踪出 tracklet
 │     → 得到 object_id → {每帧 bbox + mask} 的时空索引
 ├─ 4. 最佳帧选择:每个 tracklet 内按「面积大 + 拉普拉斯清晰度高 + mask 完整(不贴画面边缘)」
 │     打分,选最清晰的一帧
 ├─ 5. 抠图(mask 直接抠,白底 1024px)→ image-to-3D(TRELLIS)→ GLB
 │     (顺带算 embedding 存下,供审核页出"疑似同款,是否合并"候选)
 ├─ 6. 多模态识别(Claude/Qwen-VL)→ **结构化标签**(品类/子品类/颜色/材质/风格/形态特征/尺寸档)
 │     ——既是资产元数据,也是后续同款匹配的依据;与第 5 步的 3D 生成并行跑
 └─ 7. 落库:asset(含 embedding)+ track 索引 + 缩略图 + Draco 压缩后的 GLB
```

工程要点:

- **索引体积**:mask 全帧存太大。索引里 bbox 每帧都存(几十字节/帧),mask 只按关键帧存 RLE、
  中间帧用 bbox 就够(前端闪烁高亮画光点/圆角框即可,不需要像素级轮廓)。
  一个 60s/30fps 视频、10 个物体 ≈ 1800 帧 × 10 × ~40B ≈ 700KB JSON,gzip 后 ~100KB,可以整包下发给前端。
- **同物异镜头/跨视频同款**:不做自动 ReID。embedding 给审核页提供"疑似同款"候选,人工确认后合并
  (合并 = 多条 track 指向同一 asset,各自的轨迹标注都保留,溯源不丢)。
- **质量兜底**:3D 生成失败/效果差的 tracklet,资产标记 `status=rejected`,索引里仍保留 bbox
  (可以闪烁并提示"生成中/待圈选"),不影响前端交互闭环。
- **人工审核**:批量生成必有废品。做一个极简审核页(列出 缩略图+GLB 预览,通过/拒绝/改分类),
  demo 前把库里内容筛一遍——这是"资产库优先"质量可控的最后一道闸。

## 三、核心设计 B:实时识别(任意视频,1–2s)+ 圈选确认

预计算覆盖不了的场景——用户拿来平台没见过的视频——走在线链路:

```
暂停(t 秒帧) → 开集检测(Grounding DINO,单帧 0.5–2s,GPU)
            → 返回各检测框(此视频尚无标注 → 全部按「还没人圈过」显示,事实如此)
            → 检测结果写回该视频的索引缓存(lazy indexing:
              第一个暂停的人付 2 秒,之后所有人查表)

圈选某个框 → 抠图 → 多模态识别提取结构化标签(1–3s,配「分析中」动画;与库内同一套标签体系)
            → 库内标签匹配:品类硬过滤 + 标签重合度排序(embedding 可选做并列排序辅助)
            ├─ 有高分候选:弹「库里有个很像的,直接用?」用户看缩略图确认
            │   → 确认:track 挂现有 asset_id,**不重新生成 3D**
            └─ 无候选 / 用户选「生成新的」:image-to-3D + 标签 → 新资产入库
```

要点:

- **同视频判定看标注(确定性查表),跨视频复用靠标签匹配**;标签匹配只出建议、由用户或审核确认,
  建议错了的代价是用户看一眼缩略图点个"不是",不污染数据。
- **重复生成是标签不全的信号**:同一物体被生成两遍,说明当时匹配没召回。审核页把
  标签/embedding 高相似的资产对自动列为"疑似重复"候选,人工确认合并 → 标签取并集写全、
  track 重挂到保留的 asset。这是个自愈闭环,库越用匹配越准。
- 匹配排序在 hackathon 体量(百级资产)下用 标签重合度 + numpy 余弦即可,不引检索引擎。
- 检测部署:demo 时段在租用 GPU 上常驻 Grounding DINO(单帧推理显存 ~6GB,和 SAM2 同机放得下);
  平时零流量不挂机,评奖期可切 Replicate 按次调用兜底。
- **demo 反"PPT"的关键一幕**:现场拿一个从未入库的视频暂停 → 光点亮起 → 圈选 → 确认库内同款/触发生成。
  这证明 pipeline 是真自动化,不是手工摆拍。

## 四、数据模型

```
video   { video_id, title, source_url, duration, cover,
          status(unindexed/processing/indexed),
          index_source(offline/lazy) }
track   { track_id, video_id, category, t_start, t_end,
          frames: [{t, bbox}], keyframe_masks: [{t, rle}],
          best_frame_t, asset_id? }          ← asset_id 为空 = 检测到但未入库(普通光点,可圈选)
asset   { asset_id, name, space(客厅/卧室/餐厨…),
          labels: { category, sub, colors[], materials[], styles[],
                    features[], size_class },           ← 同款匹配依据;合并时取并集越写越全
          size_prior,
          embedding(f32[512/768]),                      ← 仅供排序辅助/疑似重复候选,非判定
          glb_url, thumb_url, source: {video_id, track_id, t_best},   ← 溯源
          merged_from: [asset_id…],                     ← 合并履历,原溯源全保留
          status(generating/ready/rejected), created_by(preset/user_id) }
user_library { user_id, asset_id, added_at, via(全选/单击/圈选) }
```

存储:hackathon 体量用 SQLite + 文件目录(GLB/缩略图/索引 JSON 静态托管)就够,embedding 存 SQLite BLOB
即可;现有 backend 的 `/storage` 静态托管直接复用。

## 五、在线 API(给前端联调)

在现有 FastAPI 骨架上加一组 router:

| 接口 | 说明 |
|---|---|
| `GET /api/assets?space=&category=&q=` | 资产库主入口:分类浏览 + 搜索 |
| `GET /api/assets/{id}` | 资产详情,含 `source.video_id + t_best`(前端跳原视频) |
| `GET /api/videos` / `GET /api/videos/{id}` | 视频列表/详情(含播放地址、`status`) |
| `GET /api/videos/{id}/index` | **整包时空索引**(gzip JSON),已索引视频加载时取一次,暂停查表本地完成 |
| `POST /api/videos/{id}/detect` | **实时识别**:`{t}` 或 `{t, frame(jpeg)}` → 单帧检测 → `[{bbox, category}]`,结果服务端写回索引缓存(lazy) |
| `POST /api/videos/{id}/select` | 圈选:`{t, bbox}` → 抠图+提取标签 → 标签匹配 → 返 `{candidates: [疑似同款+缩略图+分数]}`;前端让用户确认 → `POST …/select/confirm`(`use_asset_id` 挂现有资产不重新生成,或 `generate_new=true` 走生成 pipeline 返 job_id) |
| `POST /api/assets/merge` | 合并重复资产(审核用):`{keep_id, drop_id}` → 标签取并集、track 重挂、drop 方进 merged_from |
| `POST /api/library/batch-add` | 全选:`{asset_ids[]}` 加入我的素材库 |
| `GET /api/library` | 我的素材库 |
| `GET /api/jobs/{id}` | 复用现有异步 job 查询 |

前端逻辑:视频 `status=indexed` → 加载整包索引,暂停本地查表;`status=unindexed` → 暂停时调 `/detect`,
显示扫描动画等 1–2s。两种情况的光点渲染、圈选交互完全一致。

## 六、算力与成本评估(TODO #1 要落实的数字)

预算约束:总预算 1.2w–2.3w 元,服务器只是其中一部分,目标控制在 **≤2000 元**。

| 项 | 方案 | 估算(待 TODO#1 核实) |
|---|---|---|
| 离线批量(SAM2 + GroundingDINO) | 租 GPU 跑离线批量,**AutoDL 4090**(国内网络好,~2 元/h)或 RunPod($0.35–0.7/h) | 20 个视频 × ~10min/个 ≈ 4h ≈ **10–30 元** |
| 在线实时检测(demo/彩排时段) | 同一台租用 GPU 常驻 Grounding DINO(+SAM 单帧),demo+彩排 ~10h | **~20–30 元**;评奖期切 Replicate 按次(~每次几美分) |
| image-to-3D | **首选继续 fal TRELLIS**(质量已验证,按次付费);自部署 TRELLIS 仅当批量单价算下来更贵或配额受限时启用 | 300 个资产 × 每次几美分~几毛 ≈ **100–1000 元,需实测核价** |
| embedding(CLIP/DINOv2) | 和检测同机跑,增量算力忽略不计 | ~0 |
| 多模态识别 | Claude/Qwen-VL API,每资产 1 次 | 300 次 ≈ **几十元** |
| 在线服务(API + 静态托管) | 轻量云服务器 2c4g(查表/转发/匹配不需要 GPU;GPU 仅检测,按时段挂) | **~50–100 元/月** |

关键判断:常驻服务不需要 GPU,GPU 分两种用法——离线批量按小时租、在线检测只在 demo/彩排时段挂,
用完即停。这与会议"自建服务器部署开源模型"的诉求一致,但比 24h 挂一台 GPU 机便宜一个数量级。
若 fal 批量核价可接受,3D 生成不必自部署,GPU 只跑 检测/SAM2/embedding。

# PoC 实测记录

## 测试 1：image-to-3D 几何质量（2026-07-06）

**工具**：Hunyuan3D-2 官方 HuggingFace Space（免费 ZeroGPU）
**输入**：内置示例——一盏欧式落地灯/庭院灯（家居照明类，正好对口 DreamHome 场景）
**配置**：Gen Shape（仅几何）、Turbo 模式、导出 glb、目标面数 10000

### 结果

✅ **几何质量：优秀，可直接用。**
- 灯头六面玻璃灯罩 + 金属框架、顶部圆球尖顶完整还原
- 灯柱的凹槽/车削线脚、底座多层叠涩造型都清晰
- 拓扑规整，和输入图高度一致，无明显破面/空洞

⏱️ **耗时：约 2 分钟**（含 ZeroGPU 排队 + 生成 + 模型载入 viewer）。
> 这印证了技术调研里的核心风险：**免费/共享 GPU 有排队，现场 demo 必须预生成 + 缓存**，不能让评审等 2 分钟。自建/付费 API（Tripo/Meshy）或本地部署会快很多（30–60s 级）。

⚠️ **纹理（贴图）**：这个免费 Space 关闭了纹理合成（缺依赖），只出灰模几何。
> 想要带贴图的成品，需走：Tripo/Meshy 商业 API，或本地部署 Hunyuan3D 完整版（含 texture pipeline），或 TRELLIS。

### 对 DreamHome 的结论

- 「摘抄一个元素 → 变成 3D 组件」这条核心链路的**几何生成部分已验证可行**，家具/家居类质量足够撑起 demo。
- 贴图质量是决定"好不好看"的关键，需要在带纹理的方案上再单独实测一次（下一步）。
- Demo 工程上必须做**预生成 + 缓存**，规避生成延迟。

### 环境侧记（实测踩的坑，供复现参考）

- tripo3d.ai 在本机（Clash 全局代理）打不开 → 换 HuggingFace 路径。
- HF Space 的 Gradio app 套在跨域 iframe 里，浏览器自动化拿不到内部文件 input 的 ref → **直接访问 `*.hf.space` 独立域**可绕过。
- 浏览器 file_upload 工具当前版本不接受本机路径 → 自定义家具照片没能上传，改用内置示例测试。想上传自己的探家截图实测，需换支持 files 传参的上传方式或本地部署。
